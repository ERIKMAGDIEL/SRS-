1. Introducción

1.1 Propósito

Este documento de Especificación de Requisitos de Software (SRS) describe de manera detallada los requisitos funcionales y no funcionales de la aplicación Gestión Dormitorios HVU-ULV. Está dirigido al Preceptor (cliente), al equipo de desarrollo, al personal de pruebas y a los demás interesados en el proyecto.

1.2 Alcance

La aplicación tiene como objetivo centralizar y digitalizar la gestión de la disciplina y actividades del Hogar de Varones Universitarios (HVU) de la Universidad Linda Vista, incluyendo asistencia, limpieza, reportes disciplinarios y amonestaciones, a través de una app móvil para Android e iOS y una API backend conectada a SQL Server.

1.3 Definiciones, acrónimos y abreviaturas

  • HVU: Hogar de Varones Universitarios.
  • Preceptor: Responsable administrativo del HVU y cliente del proyecto.
  • Monitor: Estudiante con rol adicional de apoyo en la gestión.
  • API: Application Programming Interface.
  • REST: Representational State Transfer.
  • SRS: Software Requirements Specification.

2. Descripción general

2.1 Perspectiva del producto

La aplicación Gestión Dormitorios HVU-ULV es una solución de tipo cliente-servidor que forma parte del entorno tecnológico de la Universidad Linda Vista. El frontend se implementa como una app móvil en Flutter (Android/iOS), mientras que el backend se desarrolla en Node.js/Express y se conecta a una base de datos SQL Server institucional.

2.2 Funciones del producto

  • Registrar y consultar asistencia a cultos y actividades mediante códigos QR.
  • Registrar evaluaciones de limpieza por cuarto, con criterios y calificaciones.
  • Gestionar reportes disciplinarios (creación, consulta, aprobación).
  • Registrar y consultar amonestaciones por nivel.
  • Gestionar roles de Monitores asignados a estudiantes.
  • Permitir a los estudiantes consultar su información relevante (asistencia, limpieza, reportes, amonestaciones).

2.3 Características de los usuarios

  • Estudiante (Rol 3): Usuario principal, consulta su información y registra asistencia mediante QR.
  • Monitor (Rol 2): Estudiante promovido con permisos para registrar limpieza, generar QR y gestionar reportes.
  • Preceptor (Rol 1): Administrador del HVU, con acceso a vistas globales, asignación de roles y gestión de reportes y amonestaciones.

2.4 Restricciones

  • Uso obligatorio de SQL Server como base de datos institucional.
  • Conectividad a Internet requerida para el uso de la mayoría de las funcionalidades.
  • Compatibilidad mínima con Android 6.0 y iOS 12.0 (según lo definido en el proyecto).
  • Cumplimiento de las políticas internas de seguridad de la Universidad Linda Vista.

2.5 Suposiciones y dependencias

  • Se asume que los datos de estudiantes y preceptores son proporcionados por sistemas institucionales oficiales.
  • Se asume que el HVU proveerá infraestructura básica (red, servidor, dispositivos para pruebas).
  • El correcto funcionamiento depende de la disponibilidad del servidor de base de datos y la API backend.

3. Requisitos específicos

3.1 Requisitos funcionales

ID Nombre Descripción Prioridad
RF-01 Autenticación de usuarios El sistema debe permitir que Estudiantes, Monitores y Preceptor inicien sesión con UsuarioID y contraseña. Alta
RF-02 Registro de asistencia por QR El sistema debe permitir que los estudiantes registren su asistencia escaneando códigos QR generados por Monitores. Alta

3.2 Requisitos de interfaz externa

3.2.1 Interfaces de usuario

La aplicación debe ofrecer pantallas para:

  • Pantalla de inicio de sesión.
  • Dashboard de Estudiante, Monitor y Preceptor.
  • Pantallas para asistencia, limpieza, reportes y amonestaciones.

3.2.2 Interfaces de hardware

El sistema requiere dispositivos móviles con cámara funcional para escanear códigos QR y conexión Wi-Fi/datos.

3.2.3 Interfaces de software

  • Comunicación entre app Flutter y API REST (Node.js/Express) mediante HTTP/JSON.
  • Conexión de la API con SQL Server mediante el driver mssql.
  • Posible integración con APIs institucionales para obtener datos de alumnos.

3.2.4 Interfaces de comunicación

La comunicación entre cliente y servidor debe realizarse mediante protocolo HTTP/HTTPS en la red institucional o por Internet, según el despliegue definido.

3.3 Requisitos no funcionales

3.3.1 Rendimiento

  • El tiempo de respuesta no debe superar los 2 segundos en operaciones típicas.
  • La API debe soportar el uso concurrente de los estudiantes del HVU.

3.3.2 Seguridad

  • Las operaciones sensibles deben requerir autenticación.
  • El sistema debe validar entradas para prevenir inyecciones SQL.
  • Los roles deben controlar el acceso a funcionalidades (Estudiante, Monitor, Preceptor).

3.3.3 Usabilidad

  • La interfaz debe ser clara y entendible para estudiantes sin conocimientos técnicos avanzados.
  • Los mensajes de error deben ser descriptivos y fáciles de interpretar.

3.3.4 Fiabilidad y mantenibilidad

  • El sistema debe manejar errores de red o servidor sin perder consistencia en los datos.
  • El código debe estar modularizado para facilitar su mantenimiento y evolución.

3.4 Requisitos de datos

  • Los datos de estudiantes deben incluir al menos: matrícula, nombre completo, carrera, cuarto y estado.
  • Las tablas de limpieza, asistencia, reportes y amonestaciones deben incluir referencias a los usuarios involucrados.
  • Los datos deben respetar las políticas de privacidad y manejo de información de la institución.
