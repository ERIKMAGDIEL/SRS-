ESPECIFICACIÓN DE REQUISITOS DEL SOFTWARE (SRS)

App Gestión Dormitorios HVU-ULV
Basado en IEEE-830

1. Introducción

1.1 Propósito

Este documento SRS describe de forma completa los requisitos funcionales, no funcionales y reglas de negocio del sistema “App Gestión Dormitorios HVU-ULV”. Su propósito es servir como contrato entre el cliente (Preceptor HVU) y el equipo de desarrollo.

1.2 Alcance

El sistema permitirá gestionar asistencia a cultos, evaluaciones de limpieza, reportes disciplinarios, amonestaciones y administración de roles dentro del Hogar de Varones Universitarios (HVU). Incluye:

  • App móvil en Flutter para Android e iOS.
  • Backend en Node.js/Express.
  • Base de datos en SQL Server.

1.3 Definiciones, siglas y abreviaturas

  • HVU: Hogar de Varones Universitarios.
  • API: Interfaz de Programación de Aplicaciones.
  • REST: Representational State Transfer.
  • CRUD: Create, Read, Update, Delete.
  • UI: Interfaz de Usuario.

1.4 Referencias

  • Plantilla IEEE-830.
  • Documentación del proyecto HVU.
  • API Flutter Docs.
  • Node.js Docs.

2. Descripción General

2.1 Perspectiva del producto

La aplicación es Cliente-Servidor y se compone de:

  • Frontend: Flutter.
  • Backend REST: Node.js.
  • BD: SQL Server.

2.2 Funciones del producto

  • Registro y consulta de asistencia por QR.
  • Evaluación de limpieza (monitores).
  • Consulta de limpieza (estudiantes y preceptor).
  • Creación y consulta de reportes disciplinarios.
  • Gestión de amonestaciones.
  • Gestión de roles (asignar/quitar monitor).

2.3 Características de los usuarios

RolDescripción
Preceptor Máxima autoridad. Aprueba reportes, supervisa limpieza, gestiona roles.
Monitor Genera QR, registra limpieza, crea reportes.
Estudiante Escanea QR, consulta limpieza, consulta reportes aprobados.

2.4 Restricciones

  • BD obligatoria: SQL Server.
  • Requiere Internet.
  • Autenticación obligatoria en todas las operaciones.

3. Requisitos Específicos

3.1 Requisitos Funcionales (RF)

RF01 – Autenticación

  • RF01.1 El usuario debe iniciar sesión con UsuarioID y contraseña.
  • RF01.2 La API debe validar credenciales.

RF02 – Gestión de Asistencia

  • RF02.1 El monitor podrá generar un QR para registrar asistencia.
  • RF02.2 El estudiante podrá escanear el QR.
  • RF02.3 La API guardará la asistencia.

RF03 – Evaluación de Limpieza

  • RF03.1 El monitor podrá evaluar cuartos con criterios asignados.
  • RF03.2 La app mostrará al estudiante su última evaluación.
  • RF03.3 El preceptor verá limpieza general por pasillos.

RF04 – Reportes Disciplinarios

  • RF04.1 Monitores y preceptores pueden crear reportes.
  • RF04.2 El reporte creado por un Monitor queda “Pendiente”.
  • RF04.3 El Preceptor puede Aprobar o Rechazar.
  • RF04.4 El estudiante solo puede ver reportes “Aprobados”.

RF05 – Amonestaciones

  • RF05.1 El preceptor puede registrar amonestaciones.
  • RF05.2 El estudiante puede ver su historial.

RF06 – Gestión de Roles

  • RF06.1 El Preceptor puede asignar o quitar rol de Monitor.

3.2 Requisitos No Funcionales (RNF)

RNF01 – Rendimiento

  • Respuestas menores a 2 segundos.

RNF02 – Seguridad

  • Uso de HTTPS.
  • Validación de sesión en cada endpoint.

RNF03 – Usabilidad

  • App intuitiva y accesible.

RNF04 – Compatibilidad

  • Android 6+ / iOS 12+.

4. Reglas de Negocio

  • RN-01: Un estudiante no puede ver reportes “Pendiente” o “Rechazado”.
  • RN-02: Un monitor solo puede evaluar limpieza, no aprobar reportes.
  • RN-03: Solo el preceptor puede asignar roles.

5. Modelo de Datos (Resumen)

Tablas principales

  • Usuarios
  • Estudiantes
  • Preceptores
  • Limpieza y LimpiezaDetalle
  • Reportes
  • Amonestaciones
  • AsistenciasCultos
  • Cat_TipoCulto
  • Cat_NivelAmonestacion

7. Criterios de Aceptación

  • Todas las funcionalidades descritas en el SRS están implementadas.
  • Sin defectos críticos abiertos.
  • Pruebas UAT aprobadas por el Preceptor.
  • Despliegue final funcional (diciembre 2025).

8. Suposiciones y Dependencias

  • El servidor SQL Server está disponible.
  • El Preceptor puede validar avances.
  • Los estudiantes tienen acceso a Internet.

No obstante, la perspectiva a futuro del producto contempla su integración como un módulo o funcionalidad dentro de una aplicación más amplia y centralizada denominada UNIPAS. Se espera un nivel básico de alfabetización digital por parte de los usuarios. ● Estabilidad de las reglas de negocio: Se asume que las reglas de negocio definidas para la generación de reportes automáticos (ej: tres faltas matutinas), las políticas de asignación de permisos y los flujos de trabajo básicos se mantendrán relativamente estables durante el desarrollo del sistema. Cambios significativos en estas reglas podrían requerir modificaciones sustanciales en el sistema. ● Funcionalidad de los dispositivos de escaneo: Para el registro de asistencia, se asume que las cámaras de los smartphones o los lectores de códigos de barras utilizados por los monitores funcionarán correctamente y serán ● Entrega y funcionamiento de los lectores de códigos de barras (si aplica): Si se requiere hardware específico para el escaneo (lectores de códigos de barras dedicados), el proyecto depende de la adquisición, entrega e instalación oportuna de estos dispositivos y de su correcto funcionamiento. ● Integración con sistemas existentes (si aplica): Si el nuevo sistema necesita integrarse con otros sistemas de la universidad (ej: sistema de gestión académica para obtener datos de estudiantes), el proyecto depende de la disponibilidad y la correcta funcionalidad de las APIs o los mecanismos de integración proporcionados por esos otros sistemas. La colaboración con los equipos responsables de esos sistemas será crucial. ● Aprobaciones y retroalimentación de las partes interesadas: El avance del proyecto puede depender de la obtención oportuna de aprobaciones y retroalimentación por parte de las autoridades universitarias, los preceptores, los monitores y otros stakeholders clave en las diferentes etapas del desarrollo. ● Cumplimiento de normativas legales: El sistema depende del cumplimiento de las leyes y regulaciones de protección de datos personales vigentes en México. Cualquier cambio en estas normativas podría requerir ajustes en el sistema. ● Disponibilidad del equipo de desarrollo: El progreso del proyecto depende de la disponibilidad continua y la productividad del equipo de desarrollo. Expansión a otros dormitorios: Se pretende que la aplicación pueda extender su uso a los cuatro dormitorios de la universidad , abarcando no solo el dormitorio de varones universitarios, sino también el de mujeres universitarias y los dormitorios de nivel medio tanto para varones como para mujeres. Restricciones Hardware: Los estudiantes pueden tener diferentes sistemas operativos así que la aplicación debe ser óptima para todos ellos. Permisos: Los únicos que pueden acceder a todas las pantallas de administración es el preceptor y los usuarios a los cuales el conceda permiso en este caso los monitores. Gestión de seguridad: Los estudiantes solo podrán ver sus calificaciones, reportes o amonestaciones. Pero no podrá editarlas, además ningún otro alumnos podrán ver las calificaciones de otro solo serán visibles para el alumno. Accesos restringidos: La aplicación solo podrá ser usada para alumnos internos y que vivan dentro del dormitorio.
